# Couchbase Cluster — Проверка отказоустойчивости

## Запуск кластера

```bash
cd couchbase
docker-compose up -d
chmod +x init_cluster.sh
sleep 30
./init_cluster.sh
```

Couchbase UI: http://localhost:8091 (Administrator / password)

## Наполнение данными

Данные загружаются через контейнер `client`, который работает внутри Docker-сети и имеет прямой доступ к узлам Couchbase:

```bash
docker-compose run --rm client python3 /app/fill_data.py
```

## Кейсы проверки отказоустойчивости

### Кейс 1: Остановка 1 из 3 узлов (с репликами) — ПРОШЁЛ

Кластер настроен с `replicaNumber=2`. После остановки `couchbase-node3` узел перешёл в статус Down, но данные остались доступны с реплик на двух других узлах. Чтение и запись через Query UI продолжали работать корректно.

### Кейс 2: Failover узла через UI — ПРОШЁЛ

Выполнен graceful failover узла `172.20.0.4` через Couchbase UI. Кластер перераспределил данные, чтение и запись работали без ошибок.

### Кейс 3: Перезапуск узла и Rebalance — ПРОШЁЛ

Узел запущен обратно, выполнен Rebalance через UI. Все 3 узла вернулись в статус Healthy, данные на месте.

### Кейс 4: Остановка 2 из 3 узлов — НЕ ПРОШЁЛ

Остановлены `couchbase-node2` и `couchbase-node3`. Кластер потерял кворум, запросы через Query UI завершились с ошибкой. Полная недоступность операций.

### Кейс 5: Bucket без реплик, остановка 1 узла — НЕ ПРОШЁЛ

Создан bucket `no_replicas` с Replicas: 0, заполнен 1000 документов. После остановки одного узла часть документов стала недоступна — данные, хранившиеся на упавшем узле, потеряны без возможности восстановления.

### Кейс 6: Остановка всех узлов — НЕ ПРОШЁЛ

Все 3 контейнера остановлены. Couchbase UI недоступен, подключение к кластеру невозможно.
